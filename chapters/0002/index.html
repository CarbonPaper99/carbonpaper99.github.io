1.1 控制台、日志和输出输入

O1 控制台与信息输出

在开始本章节前，我们先认识一个简单的概念：控制台（Console）

控制台是指服务器后台运行时管理程序输出输入的系统，表现出来给管理员看到的就是开启服务器后出现的大黑框命令行。服务器的信息集中到这里。控制台可以执行除了玩家命令以外的任何一种命令，也可以观察到服务器内的信息流出入。对服务器的开发、管理和排错（Debug）有很重要的意义。

控制台属于服务器的一部分，只有服务器常开，控制台才有效。同样，控制台也可以作为服务器的最高Executor（执行者）。这里体现了控制台有两种信息管理形式，分别是输出（Output）和输入（Input）。

为增进读者理解，我们先从简单的例子讲这一对概念。

从输出讲起。 输出是最简单的。在服务器开启的过程中，控制台总是会间或地输出一些信息到后台屏幕上，这些信息来自于服务器内部。对于它们，有两种最普通的表现形式。第一种是服务器正常运行的信息流，第二种是报错的信息流。

报错信息流一般体现为以警告色标注（红色、黄色），信息内含有ERROR、Exception或Fail等字样且具有非常规的字符弹出的信息流，下面是个例子：

常规的信息流中包括事件信息流，可以看到玩家登陆、登出，玩家发出信息、使用指令、达成成就等事件的情况。举例，这是我的生存服务器内一段时间内的控制台日志：

O2 控制台与输入

控制台除了输出，也可以输入。不过控制台的输入端口以指令为基础，操作都由输入的指令完成。这里的指令同游戏内玩家使用的斜杠命令行/help or /gamemode的命令相同，只不过输入这些命令时你可以省略斜杠，你可以直接输入指令的内容和参数执行指令，下面是个例子：

> gamemode Joxes survival
... > 已设置玩家Joxes的游戏模式为 survival

这一例里，服务器控制台向服务器输入了一条指令，指令主干为gamemode，参数为Joxes和survival。已知/gamemode使用下有多种参数可以选择，具体参照的是：/gamemode <玩家名字> <游戏模式>，这条指令在控制台输入时也获得了同样的效果，故可得知：

控制台在执行指令时，具有最高权限。

然而我们不能认为控制台与玩家权限等价，下面是个例子：

> give minecraft:diamond 1
... > 正确用法：/give <玩家> <物品> <数量>

#或

>> kill
... > 正确用法：/kill <玩家>

# 或

>> tp 0 0 0
... > 正确用法：/tp <玩家> X Y Z

不难发现，在控制台输入能作用于实体的指令时，不指定对象是无法执行的，这是因为控制台本身不能算作实体，自然无法对自身进行有关实体的命令执行。所以需要记得的是：

控制台使用有实体效应的指令时，必须指定对象。

实体效应的命令有许多，例如 /give , /effect , /tp , /kill , ...

综上即为控制台的基本概念。

2.1 服务端文件的修改，参数的配置等技巧

上面关于日志信息的辨识和读取技巧提到了logs文件夹，logs文件夹只是服务端程序文件中的单个文件夹。服务端完整程序含有21个文件（夹），完整的结构如下：

- /logs #日志
- /plugins #插件 (!)
- /world #主世界的数据 (!)
- /world_nether #下界世界的数据
- /world_the_end #末地世界的数据
- eula.txt #EULA协议书
- paper.yml # paperspigot的配置文件 (!)
- spigot.yml # spigot的原生配置文件
- bukkit.yml # bukkit的原生配置文件 (!)
- ops.json # 超级管理员名单
- usercache.json # 用户的杂项数据
- server.properties # 服务器基础项参数配置文件 (!)
- help.yml # 帮助文档
- commands.yml # 自定义指令配置文件
- banned_players.json # 被封禁的玩家名单
- banned_ips.json # 被封禁的IP地址名单
- version_history.json # 历史版本
- PaperSpigot.jar # 服务端核心文件  (!)
- xxx.bat # 服务端启动脚本 (!)

在开始服务器配置之前，我们先明白一个规则：无法断定作用的文件须先保持原样，切记不要删除或随意修改，这是一个服务器开发维护时的准则。

不难从网页标示出来的高亮文字看出，整个配置文件分成两部分，一是正常的xx = xxxx的格式，另一种是以#这种井号标成灰色的文字。我们通常称这种灰色的文字为注释（description），注释是对文档的补充说明，方便开发者进行开发的一种规范。这种文本不会影响程序运作，相当于说明书。在properties文件中，注释的开头必须加井号，在Notepad++中，加了井号的注释也会被标明特殊颜色。

理解了文字意义后，我们可以开始编辑这份文件了，你可以根据上面的注释来依次修改你服务端当中文件的对应参数（方括号内是可以填入的值类型）。例如server-ip=我们可以改成server-ip=127.0.0.1，当然也可以修改成server-ip=0.0.0.0这样的地址。等号后面是参数的值，等号前面是参数键（即key）。配置当中也时常会出现是否值，true即为是，false即为否。具体可见本章下节

以上是properties语言的配置规范，它是我们服务端配置规则中的一角。

O2 YAML 配置语言

除了properties文件格式，我们还将在服务器中认识到一种最常规的配置文件语言：YAML

YAML的文件以.yml结尾，这种文件在服务端中有很多，例如spigot.yml和bukkit.yml，YAML特点在于简洁和适应性强，因而在进行配置时，YAML总是最优选。

举个例子，下面是一段YAML语言配置文件的规范：

name: "CarbonPaper99"
sex: "male"
age: 0
isStudent: true
lore:
- "the first line"
- "the second line"

#我是YAML的注释
#YAML语言同properties一样以井号开头进行注释。

在YAML文件中，冒号前面的内容称作参数键（key），冒号后面的内容称作参数值（value）。由此不难看出YAML依然是和Properties相近的标记语言，都是以“键:值”为基础进行标记的语法。这种语法中，数值有四种基本类型：

  • 以整数或小数组成的数字值（例如1、4、1.5、3.1415926等值）
  • 以true/false标记出“是/否”的布尔值（boolean）（true代表是，false代表否）
  • 以双引号包围的字符串（例如"OKay","CarbonPaper"）
  • 以横杠开头，换行代表下一行，可以多行叠加的字符串组或者是数组（数组参数键: - "数组的第一行"\换行 - "数组的第二行"）
YAML的参数键:YAML的参数值

由于YAML依然以英文为基础，所以所有关于语法有关的字符标点都应当由英文字符代替，不可使用中文的全半角字符。

请注意，一般的YAML文件中，参数键和参数值除了括号外还需一个空格隔开，如： key[冒号：][空格]value

O3 JSON文件

JSON语言是一种用于标记配置的语言，JSON文件的后缀名即为 .json在Minecraft中，JSON格式的文件用来标记配置的情况屡见不鲜。JSON文件的行文较为复杂，在这里仅需认识，后文将多加讲解。摘抄一部分的JSON文件行文格式如下：

// 数组JSON结构：

			"firstName": "Brett",

编辑完毕后，通常要检查文件的编码（Encode）是否吻合。计算机根据不同的编码对非英文字符进行转化识别。如果编码出现问题，配置文件会出现非英文字符（包括中文及其标点）变成乱码，而导致前功尽弃。我们在选择编码时可以对照下表进行选择。（更改文件编码可以在Notepad++中点击编码，再点击转向UTF-8或ANSI）

1.9以前 ANSI UTF-8
1.9至如今 UTF-8 UTF-8

对照并更改自己文件的编码，随后重启服务器即可生效。（调试配置文件后重启服务器需要养成备份习惯）

2.3 超级管理员和服务器玩家操作

O1 超级管理员

即OP（亦称为Operator，操作员），是一种高于普通玩家的权限等级，此类玩家可以使用大量非常规指令，如服务器排错、作弊、修改等类型指令。OP玩家所能获取到的最大权限限度可以在server_properties.json中进行设定。将一名玩家设定为操作员，输入：

/op <玩家>

同样，撤销一名玩家的操作员权限可以：

/deop <玩家>

O2 玩家操作、黑名单

一般的，对于玩家的操作可以有如下选项：

  • 如果你想踢出某个玩家，你可以使用/kick <玩家> [理由]命令使玩家退出本局游戏，但玩家还可以继续进入游戏。

  • 如果你想将某个玩家拉入黑名单（即ban某个玩家），可以使用/ban <玩家> [理由]这个指令，玩家和他的UUID将会被记录到banned_players.json中。玩家将永远不可加入这个服务器。

  • 如果你想彻底封禁该玩家，可以直接将该玩家的IP地址拉入黑名单（不建议这样做，因为如果使用外网映射，玩家将统一从127.0.0.1端口进入，这将会封杀所有玩家），这个IP地址下的所有设备将无法进入服务器，具体可使用/ban-ip <地址> [理由]封杀，IP地址会记入到banned_ips.json中

解封一个IP地址或者是玩家，可以使用/pardon <地址/玩家>

2.4 世界的开关和更换

O1 多世界开关

原版Minecraft服务端（以1.12为例）默认带有三个世界，即主世界、下界即末地。要关闭其中的世界防止玩家进入，可以在以下的文件中分别找到下列配置项：

# 服务端 bukkit.yml 文件
# 针对末地世界的开关选项
allow-end: false

# 服务端 server.properties 文件
# 针对下界世界的开关选项
allow-nether=false

上述开关可以自定义配置，世界选项关掉之后，该世界的文件可以被删除而不会恢复。

O2 多世界更换

对不同世界的文件予以更换，只需要在服务器停止的情况下将原有的世界文件删除，并转移新的世界文件进入即可。

请注意，更换的世界并不适用于所有服务端，部分世界可能因为兼容问题而被服务端选择覆盖，请注意备份。

2.4 本章 FAQ

Q1 如何关闭服务器？

A1 控制台输入stop即可完成

Q2 如何重启服务器？

A2 重启服务器有两种方法，第一种是使用stop后再重新启动服务器，这种情况最保险。第二种是使用reload命令，这个命令可能在一些情况下会导致服务器崩溃。

Q3 关于服务端的其他配置文件的配置技巧，为什么没有提到？

A3 因为这些服务端文件都大同小异，所有服务端文件的更改方法和完整注释都可以在Minecraft-WIKI找到，本站只是举例修改。你也可以在本站找到所有服务端注释过的文件，你可以根据这些注释修改文件。

Q4 如何升级服务器？

A4 可以通过上一章提到的下载地址下载更高版本的服务端核心文件，同现有服务端核心文件更换，服务端文件一般可以向上替换（高版本替换低版本），但不可以向下（低版本替换高版本）。
