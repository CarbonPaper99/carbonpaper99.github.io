1.1 控制台、日志和输出输入

O1 控制台与信息输出

在开始本章节前，我们先只认识一个简单的概念：控制台（Console）

控制台是指服务器后台运行时管理程序输出输入的系统，表现出来给管理员看到的就是开启服务器后出现的大黑框命令行。服务器的信息集中到这里。控制台可以执行除了玩家命令以外的任何一种命令，也可以观察到服务器内的信息流出入。对服务器的开发、管理和排错（Debug）有很重要的意义。

服务器和控制台是绑定在一起的，只有服务器常开，控制台才有效。同样，控制台也可以作为服务器的最高Executor（执行者）。这里体现了控制台有两种信息管理形式，分别是输出（Output）和输入（Input）。

好，控制台的概念我们基本了解完了。在知道了控制台的输出输入双面性。为了理解这个输入和输出的概念，我们举几个例子来讲，增进读者的理解。

先讲输出吧，输出是最简单的。在服务器开启的过程中，控制台时常会弹出一些信息到控制台窗口上，这些信息来自于服务器内部，有两种最普通表现形式。第一种是服务器正常运行的信息流，第二种是报错的信息流。

报错信息流一般体现为以警告色标注（红色、黄色），信息内含有ERROR或Fail等字样且具有非常规的字符弹出的信息流，下面是个例子：

O2 控制台与输入

控制台除了输出，也可以输入。不过控制台的输入端口以指令为基础，操作都由输入的指令完成。这里的指令同游戏内玩家使用的斜杠命令行/help or /gamemode的命令相同，只不过输入这些命令时你可以省略斜杠，你可以直接输入指令的内容和参数执行指令，下面是个例子：

> gamemode Joxes survival
... > 已设置玩家Joxes的游戏模式为 survival

这一例里，服务器控制台向服务器输入了一条指令，指令主干为gamemode，参数为Joxes和survival。既然都知道/gamemode使用下有多种参数可以选择，具体参照的是：/gamemode <玩家名字> <游戏模式>，这条指令在控制台输入时也获得了同样的效果，故可得知：

控制台在执行指令时，具有最高权限。

但是我们可不可以认为控制台拥有和玩家等效的权限模式呢？不可以这样说。以下是一个例子：

> give minecraft:diamond 1
... > 正确用法：/give <玩家> <物品> <数量>

#或

>> kill
... > 正确用法：/kill <玩家>

# 或

>> tp 0 0 0
... > 正确用法：/tp <玩家> X Y Z

我们发现，如果在控制台直接只输入give指令并加上钻石的参数，是无法执行的。但在游戏中，倘若你要给自己一个钻石，使用这个指令可以完美运行。要理解这个就先理解这些命令的含义，我们知道这些命令既然作用于玩家上，那对玩家肯定有影响。现在控制台输入和玩家一样的命令，按道理控制台也应该受到相同的作用的，

可是，控制台属于玩家吗？

理解到这一步就够了：控制台并不是玩家，所以和玩家使用命令有小小的不一样，概括起来就是：

控制台使用有实体效应的指令时，必须指定对象。

综上，控制台的基础概念讲完。

2.1 服务端文件的修改，参数的配置等技巧

上面关于日志信息的辨识和读取技巧提到了logs文件夹，logs文件夹只是服务端程序文件中的单个文件夹。服务端完整程序含有21文件，完整的结构如下面：

- /logs #日志
- /plugins #插件 (!)
- /world #主世界的数据 (!)
- /world_nether #下界世界的数据
- /world_the_end #末地世界的数据
- eula.txt #EULA协议书
- paper.yml # paperspigot的配置文件 (!)
- spigot.yml # spigot的原生配置文件
- bukkit.yml # bukkit的原生配置文件 (!)
- ops.json # 最高级管理员名单
- usercache.json # 用户的杂项数据
- server.properties # 服务器基础项参数配置文件 (!)
- help.yml # 帮助文档
- commands.yml # 自定义指令配置文件
- banned_players.json # 被封禁的玩家名单
- banned_ips.json # 被封禁的IP地址名单

- PaperSpigot.jar # 服务端核心文件  (!)

在开始服务器配置之前，我们先明白一个规则：不懂的文件先保持原样，切记不要删除或随意修改，这是一个服务器维护时的准则。

不难从网页标示出来的高亮文字看出，整个配置文件分成两部分，一是正常的xx = xxxx的格式，另一种是以#这种井号标成灰色的文字。我们通常称这种灰色的文字为注释（description），注释是对文档的补充说明，方便开发者进行开发的一种规范。

我们想象用裱花袋在糕点上裱上字体，既能吃也能看

这样理解下来，就可以明白出：这种文本不会影响程序运作，相当于说明书。在properties文件中，注释的开头必须加井号，在Notepad++中，加了井号的注释也会被标明。

理解了注释后，我们可以开始编辑这份文件了，你可以根据上面的注释来依次修改你服务端当中文件的对应参数（方括号内是可以填入的所有值）。例如server-ip=我们可以改成server-ip=127.0.0.1，当然也可以修改成server-ip=0.0.0.0这样的地址。等号后面是参数的值，等号前面是参数键（即key）。配置当中也时常会出现是否值，true即为是，false即为否，这种数值称作布尔值，这是一种比较常见的

以上是properties语言的配置规范，它是我们服务端配置规则中的一角。

O2 YAML 配置语言

除了properties文件格式，我们还将在服务器中认识到一种最常规的配置文件语言：YAML

YAML的文件以.yml结尾，这种文件在服务端中有很多，例如spigot.yml和bukkit.yml，YAML有个特点就是简洁和适应性强，所以在进行配置的时候，YAML总是最优选。

举个例子，下面是一段YAML语言配置文件的规范：

name: "CarbonPaper99"
sex: "male"
age: 0
isStudent: true
lore:
- "the first line"
- "the second line"

#我是YAML的注释
#YAML语言同properties一样以井号开头进行注释。

在YAML文件中，冒号前面的内容称作参数键（key），冒号后面的内容称作参数值（value）。由此不难看出YAML依然是和Properties相近的标记语言，都是以“键:值”为基础进行标记的语法。这种语法中，数值有四种基本类型：

  • 以整数或小数组成的数字值（例如1、4、1.5、3.1415926等值）
  • 以true/false标记出“是/否”的布尔值（boolean）（true代表是，false代表否）
  • 以双引号包围的字符串（例如"OK","CarbonPaper"）
  • 以横杠开头，换行代表下一行，可以多行叠加的字符串组或者是数组（数组参数键: - "数组的第一行"\换行 - "数组的第二行"）

由于YAML依然以英文为基础，所以所有关于语法有关的字符标点都应当由英文字符代替，不可使用中文的全半角字符。

2.3 超级管理员和服务器玩家操作

O1 超级管理员

即OP（亦称为Operators，操作员），是一种高于普通玩家的权限等级，OP玩家可以使用大量非常规指令，如服务器排错、作弊、修改等类型指令。OP玩家所能获取到的最大权限限度可以在server_properties.json中进行设定。将一名玩家设定为OP玩家，输入：

/op <玩家>

同样，撤销一名玩家的OP权限可以：

/deop <玩家>

O2 玩家操作、黑名单

在管理玩家时，如果你想踢出某个玩家，你可以使用/kick <玩家> [理由]命令使玩家退出本局游戏，但玩家还可以继续进入游戏。

如果你想将某个玩家拉入黑名单（即ban某个玩家），可以使用/ban <玩家> [理由]这个指令，玩家和他的UUID将会被记录到banned_players.json中。玩家将永远不可加入这个服务器。

如果你想沙仁猪心，可以直接将某个IP地址拉入黑名单（不建议这样做，因为如果使用外网映射，玩家将统一从127.0.0.1端口进入，这将会封杀所有玩家），这个IP地址下的所有设备无法进入服务器，具体可使用/ban-ip <地址> [理由]封杀，IP地址会记入到banned_ips.json中

解封一个IP地址使用/pardon <地址/玩家>

2.4 本章 FAQ

Q1 如何关闭服务器？

A1 控制台输入stop即可完成

Q2 如何重启服务器？

A2 重启服务器有两种方法，第一种是使用stop后再重新启动服务器，这种情况最保险。第二种是使用reload命令，这个命令可能在一些情况下会导致服务器崩溃。

Q3 关于服务端的其他配置文件的配置技巧，为什么没有提到？

A3 因为这些服务端文件都大同小异，所有服务端文件的更改方法和完整注释都可以在Minecraft-WIKI找到，本站只是举例修改。你也可以在本站找到所有服务端注释过的文件，你可以根据这些注释修改文件。
